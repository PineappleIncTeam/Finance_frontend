# Схема кэширования PWA в FREEnance

## 1. Выбранная схема кэширования

Для обеспечения баланса между производительностью и актуальностью финансовых данных выбрана следующая гибридная стратегия:

### Стратегии по типам ресурсов:
*   **Статические ассеты (Static Assets):** `CacheFirst`
    *   *Ресурсы:* JS, CSS, шрифты (woff2), изображения (png, webp, svg), иконки, manifest.json.
    *   *Логика:* Загрузка из кэша для максимальной скорости. Обновление только при изменении хэша файла.
    *   *Срок хранения:* 30 дней.
*   **Публичные страницы (Public Pages):** `StaleWhileRevalidate`
    *   *Маршруты:* `/blog`, `/blogPage`, `/aboutUs`, `/aboutApp`.
    *   *Логика:* Показываем старую версию из кэша мгновенно, в фоне скачиваем обновление.
    *   *Срок хранения:* 24 часа.
*   **Страницы авторизации и восстановления (Auth/Recovery):** `NetworkOnly`
    *   *Маршруты:* `/login`, `/signUp`, `/changePassword`, `/newPassword`, `/activate`.
    *   *Логика:* Никогда не кэшируются. Требуют живого соединения для безопасности.
*   **API запросы (Private/Fintech API):** `NetworkOnly`
    *   *Маршруты:* Все запросы к `/api/v1/`.
    *   *Логика:* Финансовые данные (баланс, транзакции) должны быть на 100% актуальными. Кэширование на уровне Service Worker запрещено, чтобы избежать отображения неверных данных.

## 2. Другие рассмотренные варианты

Ниже приведены стратегии, которые рассматривались в процессе проектирования:

| Стратегия | Описание | Подходит для PFM? | Статус |
| :--- | :--- | :--- | :--- |
| **NetworkFirst** | Сначала сеть, если нет — кэш. | Опасно для баланса. | Рассматривалась |
| **CacheOnly** | Только кэш. | Только для шрифтов. | Рассматривалась |
| **NetworkOnly** | Строго через сеть. | **Идеально для API.** | **Выбрано для API** |
| **StaleWhileRevalidate** | Кэш + обновление в фоне | Хорошо для контента. | **Выбрано для Блога** |
| **CacheFirst** | Кэш, если нет — сеть. | Хорошо для статики. | **Выбрано для Assets** |

## 3. Технические особенности реализации

### Почему используется Webpack вместо Turbopack?
На текущем этапе сборка проекта выполняется через Webpack (`next build --webpack`), несмотря на то, что Next.js 16 поддерживает Turbopack.

**Причины:**
1.  **Совместимость Serwist:** Библиотека Serwist (и лежащий в ее основе Workbox) использует плагины Webpack для автоматической генерации манифеста прекэширования и бандлинга Service Worker'а.
2.  **Стабильность:** Поддержка Turbopack в Serwist на данный момент находится в экспериментальной стадии. Для обеспечения надежной работы PWA-функционала и корректной сборки `sw.js` требуется стабильный движок Webpack.
3.  **Ограничения плагинов:** Многие продвинутые функции Workbox (например, `ExpirationPlugin`) требуют специфической обработки на этапе сборки, которая пока не полностью реализована в Turbopack.

*   [next.config.ts](file:./next.config.ts) — инициализация Serwist.
*   [sw.ts](file:./src/sw.ts) — описание правил кэширования.
